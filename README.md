# Fenrua Labs: Protocol Infrastructure

Fenrua Labs builds privacy-preserving protocol infrastructure for verifiable AI automation.

The current direction is the **Fenrua BlackBox Protocol**: an evidence-first framework for making autonomous AI activity inspectable without exposing the confidential infrastructure that produces and protects it. Fenrua treats AI execution evidence like a protected flight recorder: public where evidence should be reviewed, private where infrastructure must remain protected.

Our work focuses on mathematical transparency, evidence-based security, protocol infrastructure, P-521 limb arithmetic, bounded signed observations, and source-bound technical review.

## What Exists Today

**Public evidence interface**  
Fenrua Web provides the public review surface for technical readers: evidence records, trust boundaries, release verification, local validation paths, operations boundaries, and reviewer-oriented navigation.

**Fenrua Kernel direction**  
Fenrua is focused on the **Fenrua Kernel** — an audit-sealed P-521 limb-based primitive designed for verifiable agent execution, deterministic validation, and future proof-oriented protocol work.

**Protected two-chain private mesh**  
Fenrua includes a protected two-chain private-mesh chain environment with more than **850,000 blocks** produced across the environment. The private mesh is not exposed as public RPC, a trading network, a customer wallet, or a financial product. Public surfaces expose only bounded, read-only observation context where appropriate.

**Source-bound validation**  
The public repository and release process are evidence-bound. Public claims, generated routes, release records, local validation commands, and public/private limitations are kept aligned so professional reviewers can inspect the protocol surface without receiving protected infrastructure access.

## The Fenrua Standard

**Primitive-First**  
High-performance, low-level architecture designed to reduce unnecessary legacy overhead and support verifiable execution.

**Audit-Sealed**  
Protocol advancements are developed with public security logs, evidence records, release history, and Genesis-linked technical context.

**Evidence-Backed**  
Critical arithmetic and protocol components are supported by differential testing, reproducible validation, source-bound records, and explicit limitations.

**Blackbox-Safe**  
Fenrua exposes the correct evidence without exposing private mesh topology, provider routes, keys, endpoints, treasury details, activation profiles, or confidential operational wiring.

## Protocol Direction

Fenrua’s long-term direction is to support privacy-preserving AI automation through:

- tenant-scoped evidence projections;
- encrypted event custody;
- replay-resistant nullifiers;
- P-521 limb-based verification primitives;
- signed and bounded observations;
- tenant logical blocks;
- future zero-knowledge ingress validation;
- hidden global aggregation;
- strict public/private infrastructure separation.

The core principle is simple:

> Reviewers should be able to inspect evidence without being handed the infrastructure.

## Public Boundary

Fenrua is not a consumer social platform, public exchange, token launch, investment product, staking system, yield product, customer wallet, or public financial-services interface.

Where commercial settlement is relevant, it is treated as ordinary business payment and accounting infrastructure. Public records must not be read as tokenomics, financial return, trading access, or customer custody.

### Distributed Codex Workflow

Fenrua was not built in a single linear Codex thread. The project used a distributed, task-specific workflow across multiple Codex/GPT-5.6 sessions because the protocol North Star evolved during the build period.

Two major recovered build-history anchors were:

- **Sol** — approximately **8,015 lines** of latest-vision project work.
- **Terra** — approximately **6,300 lines** of major project-development work.

Separate sessions were used for repository implementation, architecture review, validation planning, public evidence copy, department-style review, compliance-safe wording, protocol roadmap alignment, and submission preparation.

The project README and submission packet document this broader workflow rather than treating one thread as the full source of project development. Sol and Terra are the primary build-history anchors; Luna is not treated as a primary project anchor.
